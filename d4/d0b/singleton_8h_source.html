25 Singletons
26 (Documentation by Mick)
27 
28 A singleton is a single instance of a class which is dynamically
29 allocated just once.
30 
31 To make a class such as CStats a singleton class, you do this:
32 
33 
34  class CStats
35  {
36  // Normal Class definition goes here...
37  ...
38 
39  // Constructor and destructor MUST BE PRIVATE, for it to be a singleton
40  private:
41  CStats( void );
42  ~CStats( void );
43 
44  // Macro to include the singleton code
45  DeclareSingletonClass( CStats );
46  }
47 
48 This macro just places some static functions in the class, and a static pointer to
49 the single instance of class itself (initially NULL)
50 
51 You can then initialize the singleton by the line
52 
53  Spt::SingletonPtr< CStats > p_stats( true );
54 
55 This is somewhere at a high level in the code (currently they are all in main())
56 
57 The original design of our singletons had in mind that there would be reference counting
58 of the references tot he singleton, and when it was no longer referenced, it would
59 delete itself. So, in many places in the code, the singletons are reference by creating a
60 "Spt::SingletonPtr", which is a smart pointer to the instance of the singleton.
61 
62 For example:
63 
64  Spt::SingletonPtr< CStats > p_stats;
65  p_stats->Update();
66 
67 In the ultimate version of this idea, the game would be comprised of many modules,
68 systems and subsystems. Each would be a singleton. Each could be dependent on
69 other singletons, but more that one module could depend on a particular singleton,
70 so you coudl not rely on a module to de-initialize the singletons it needed
71 as they might be needed elsewhere.
72 
73 So, each module would include a Spt::SingletonPtr for each of the modules that it
74 needed, so when a module was created it would increase the reference on the singletons
75 and create any that did not exists. Then when it shut down, the pointers would automatically
76 go out of scope, and any singleton that was not referenced would automatically be deleted,
77 so there would never be more systems active than there needed to be.
78 
79 However, it turned out that this was generally overkill. Most systems were initialized at the start,
80 and those that were not, we wanted to have control over the initialization of them.
81 The use of smart pointers here also obfuscated the code, and introduced a significant
82 execution overhead that was totally unnecessary.
83 
84 So, the prefered method to access a singleton now is:
85 
86  CStats::Instance()->Update();
87 
88 This simply references the instance directly (as it's inline), and so
89 does no addition processing. It's easier to type as well.
90 
91 
92 */

273 #define DeclareSingletonClass(_T) \
274 private: \
275  static _T* sSgltnInstance( bool create = false ); \
276  static void sSgltnDelete( void ); \
277  \
278  static _T* sp_sgltn_instance; \
279  static uint s_sgltn_count; \
280 public: \
281  inline static void Create(void) {sp_sgltn_instance = new _T;} \
282  inline static _T* Instance(void) {return sp_sgltn_instance;} \
283  inline bool Initilized(void) {return (sp_sgltn_instance!=NULL);} \
284  \
285  friend class Spt::SingletonPtr< _T >; \
286 
287 
288 #define DefinePlacementSingletonClass(_T, _P, _N ) \
289  \
290 _T* _T::sp_sgltn_instance = NULL; \
291 uint _T::s_sgltn_count = 0; \
292  \
293 _T* _T::sSgltnInstance( bool create ) \
294 { \
295  \
296  \
297  if ( !sp_sgltn_instance && create ) \
298  { \
299  sp_sgltn_instance = new (_P) _T; \
300  } \
301  \
302  Dbg_AssertType( sp_sgltn_instance, _T ); \
303  \
304  ++s_sgltn_count; \
305  return sp_sgltn_instance; \
306 } \
307  \
308 void _T::sSgltnDelete( void ) \
309 { \
310  \
311  \
312  Dbg_MsgAssert(( s_sgltn_count > 0 ),( "Reference count imbalance" )); \
313  \
314  if ( s_sgltn_count > 1 ) \
315  { \
316  s_sgltn_count--; \
317  return; \
318  } \
319  \
320  delete sp_sgltn_instance; \
321  sp_sgltn_instance = NULL; \
322  s_sgltn_count = 0; \
323 } \
