60 // On a regular development build, the memory usage is very different
61 // to a "final=" build. Firstly there is the extra debug code, mostly assertions,
62 // but also memory blocks have a 32 byte rather than 16 byte header
63 // and many data structures have extra fields for debugging purposes
64 // this all means that the game will not fit in 32MB, so to fix this, the
65 // script heap is placed in "debug" memory (>32MB)
66 // Just that alone would result in there seeming to be MORE memory avaialbe in the normal build
67 // so to conteeract that, we allocate a block off the bottom_up heap,
68 // The size of this block is SCRIPT_HEAP_SIZE-DEBUG_ADJUSTMENT
69 // So you need to adjust DEBUG_ADJUSTMENT so the amount of free memory
70 // on the main heap is the same in a development build
71 // and in a "final=" cd build, for example, if you had for CD final build:
72 //
73 // Name Used Frag Free Min Blocks
74 // --------------- ----- ----- ---- ------ ------
75 // Top Down: 3942K 0K 894K 884K 3933
76 // BottomUp: 23275K 23K 894K 884K 10432
77 //
78 // and on a regular build:
79 //
80 // Top Down: 4004K 0K 579K 523K 3933
81 // BottomUp: 22115K 23K 579K 523K 10601
82 //
83 // Then you see that the regular build is underreporting the actual amount of free memory
84 // and you should in theory add (894-579) = 315K
85 // In practice it's a good idea to not push things right to the wire, as memory
86 // usage can vary in differing situation (specifically when the number of blocks vary,
87 // like a large number of small allocations vs a small number of large allocations.)
88 // Since the above was from a worst case situation (NJ), I feel that 200K would be appropiate
89 
90 #define DEBUG_ADJUSTMENT (1126400 + 200000 + 300000 + 100000 + 350000 ) // (1126400) // difference in free memory for "final=" vs debug build
